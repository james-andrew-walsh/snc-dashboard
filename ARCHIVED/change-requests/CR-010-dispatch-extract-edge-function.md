## Problem

The dispatch extraction pipeline currently depends on LlamaParse (a third-party AI service) and local Python scripts running on James's Mac mini. This creates two problems: (1) LlamaParse is a fragile dependency — if the service changes output format, the pipeline breaks; (2) the Mac mini dependency means the system cannot be transferred to SNC as a self-contained Supabase project.

The dispatch PDFs are structured text reports generated by SNC's dispatch software. They are NOT scanned documents — they are database report exports with consistent, machine-readable formatting. Direct text extraction produces clean, parseable output without any AI inference.

---

## Solution

Build a Supabase Edge Function (`dispatch-extract`) that:
1. Reads a dispatch PDF from the `dispatcher_reports` storage bucket
2. Extracts text using `npm:pdf-parse` (works in Deno)
3. Parses the structured text into jobs, foremen, and equipment assignments
4. Writes the parsed data to the existing Supabase tables
5. Is idempotent — re-processing the same PDF produces identical results

### Requirements

1. **PDF text extraction in Deno**: Use `npm:pdf-parse` to extract raw text from the PDF. No AI/inference — pure text extraction.

2. **Deterministic parser**: Parse the extracted text into structured data. The PDF format is consistent:
   - Job headers: `NNNNN: JOB NAME NNNNN: JOB NAME` (doubled on same line) or `NNNNN: JOB NAME(cont.)`
   - Resource lines: `HH:MM  AM  HH:MM  PM[indent]CODEDESCRIPTION` all on one line
   - Equipment lines have `|__` indent prefix; foreman lines do not
   - Equipment descriptions start with type prefix: `LD,`, `EX,`, `BL,`, `TK,`, `TR,`, `MS,`, `RL,`, `DZ,`, `BX,`, `AT-BK,`, `AT-FK,`, `AT-CP,`, `SUB,`, `R1,`–`R9,`
   - Personnel descriptions end with role: `- FOREMAN`, `- OE-4MAN`, `- LAB-4MAN`, `- SUPT`, `- LABORER`, `- OPERATOR`, `- CEMENT MASON`, etc.
   - Report date is on page 1: `Daily Schedule Report\n04/17/2026`
   - Daily Location Notes appear between job header and first resource line
   - `Rentals` sections appear after main equipment and contain duplicate entries (skip these)
   - Continuation pages have `(cont.)` suffix on job header

3. **Alt-code extraction**: The two-letter equipment type prefix (e.g., `LD`, `EX`, `BL`, `TK`) must be extracted from the description and stored as `alt_code` on both `dispatch_equipment_assignments` and `reconciliation_results`. Format: `LD 7707`, `EX 9833`. This is the display code SNC uses operationally.

4. **Daily notes extraction**: The "Daily Location Notes" text block for each job must be captured and stored in `dispatch_jobs.daily_notes`. This text can be multi-line and may contain crew assignments, location info, contact info, cost codes, and notes about equipment being billed by other foremen. Store the full text as-is.

5. **Schema migration**: Add the following columns:
   - `dispatch_reports.pdf_path` (TEXT) — storage bucket path, e.g. `dispatcher_reports/2026-04-17.pdf`
   - `dispatch_equipment_assignments.alt_code` (TEXT) — e.g. `LD 7707`
   - `reconciliation_results.alt_code` (TEXT) — e.g. `LD 7707`

6. **Idempotent upsert**: For a given `report_date`:
   - Delete existing child rows (dispatch_jobs, dispatch_foremen, dispatch_equipment_assignments, dispatch_operators, dispatch_laborers) for that report_id
   - Re-insert from the freshly parsed PDF
   - Do NOT touch reconciliation_results (those are produced by a separate reconciliation step)
   - Update `dispatch_reports.status` to `extracted`

7. **Invocation modes**:
   - **Process all unprocessed**: GET/POST to the function with no body → scans bucket for PDFs, finds ones without a matching `extracted` report, processes them
   - **Process specific file**: POST with `{ "filename": "2026-04-17.pdf" }` → processes just that one
   - Accept any PDF filename; extract the report date from the PDF content, not the filename

8. **Code/description splitting**: In pdf-parse output, the equipment code and description are concatenated without a space separator (e.g., `8092LD, 24 JD 544P WHEEL LOADER`). Split by searching for the first equipment type prefix in the string. Everything before the prefix is the code; everything from the prefix onward is the description.

9. **Foreman code extraction**: Similarly, foreman codes and names are concatenated (e.g., `CHAJASCHAPMAN, JASON- OE-4MAN`). Split by finding the first comma in the string — the code is everything before the surname that contains the comma. Foreman codes are typically 6-8 uppercase letters (e.g., `CHAJAS`, `DOUGRE`, `MORJOH`).

### Technical Approach

- Edge function location: `core/supabase/functions/dispatch-extract/index.ts`
- Use `npm:pdf-parse@1.1.1` for PDF text extraction (confirmed working in Deno 2.7)
- Use `npm:@supabase/supabase-js@2` for database operations (same pattern as existing edge functions)
- Download PDF from storage bucket using Supabase Storage API
- Parser should be a separate module (`parser.ts`) for testability
- Follow the same patterns as the existing `telemetrics-sync` and `reconciliation` edge functions in `core/supabase/functions/`

### Reference: pdf-parse output format

The pdf-parse library produces text where each resource entry is a SINGLE LINE:
```
07:00  AM  04:00  PM     |__ 8092LD, 24 JD 544P WHEEL LOADER
07:00  AM  04:00  PMCHAJASCHAPMAN, JASON- OE-4MAN
```

This is different from PyMuPDF which puts times, codes, and descriptions on separate lines. The parser must handle the concatenated single-line format.
